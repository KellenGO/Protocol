# Protocol 产品规格说明（PRODUCT_SPEC）

## 1. 项目名称

**Protocol**

暂定中文名：**自控协议系统**

一句话定位：

> 一款将 CTDP 与 RSIP 方法论软件化的离线优先桌面应用，用于帮助用户把“自控”从模糊意志问题，转化为可触发、可执行、可记录、可回滚、可迭代的协议系统。

---

## 2. 产品不是什么

Protocol **不是**：

- 普通番茄钟
- 普通待办清单
- 普通习惯打卡 App
- 以积分、徽章、排行榜为核心的游戏化工具
- 单纯做“记录学习时长”的统计工具

这些功能中有些可能作为辅助元素出现，但它们不是产品核心。

Protocol 的核心不是“提醒用户要努力”，而是：

> 为用户建立一套能够在关键行为节点上产生约束力，并可在长期生活中递归迭代的自控协议。

---

## 3. 理论来源与产品化目标

Protocol 基于两套方法论：

### 3.1 CTDP：链式时延协议

CTDP 解决的是**单次任务尺度**上的三个问题：

1. 启动困难
2. 中途放弃
3. 规则被“下不为例”逐渐侵蚀

它通过三项机制实现：

- **神圣座位原理**：将某个标志性触发动作与“必须进入专注任务”绑定，借助任务链的累计价值形成约束力。
- **下必为例原理**：每次争议行为都必须被正式判决为“违规导致链断裂”或“永久允许并写入判例”，防止规则在模糊例外中腐蚀。
- **线性时延原理**：通过“预约启动”机制，把“现在开始”的高阻力，改造成“稍后必须开始”的低阻力承诺。

### 3.2 RSIP：递归稳态迭代协议

RSIP 解决的是**长期生活稳态**上的问题：

1. 为什么 CTDP 在有 DDL、有压力、有目标时特别强，但在无压力、状态散乱、生活负稳态中会失灵
2. 如何不依赖“当下意志力”，而是通过一组轻量、局部最优、能互相保护的小定式，逐步改变生活整体稳态
3. 如何允许失败、回滚与重建，把崩塌视为系统调试，而不是单纯失败

RSIP 的核心产品化概念包括：

- **国策 / 定式节点**：一个明确、低阻力、可执行、能改善局部稳态的小规则
- **国策树**：父节点的稳定，为子节点提供可生存的条件；整棵树共同改变用户的宏观稳态
- **节点点亮 / 熄灭**：某定式当前是否处于生效状态
- **回滚**：父节点崩塌时，依赖其存在的子节点同步熄灭
- **国策组**：将多个相关节点打包，允许一定容错，避免小意外造成系统全盘崩塌
- **失败—强化—再挑战**：每次崩塌都用于发现设计缺陷，再调整规则继续迭代

## 4. 产品总体结构

Protocol 长期可分为两个大模块：

1. **CTDP 执行系统**
   - 管理主链、预约链与专注任务
   - 管理判例库
   - 记录行为执行历史

2. **RSIP 稳态系统**
   - 管理国策树
   - 管理节点点亮与熄灭
   - 管理国策组容错
   - 管理崩塌、回滚与迭代日志

### 4.1 阶段性策略

第一阶段应优先完成 CTDP 模块，原因是：

- CTDP 的交互边界更清晰
- 能更快做出可真实使用的 MVP
- 用户每天都能使用
- 其核心流程更适合先验证产品形态

RSIP 作为第二阶段模块，在第一阶段先保留导航入口与数据架构余地，不急于完整实现。

## 5. 目标用户

### 5.1 主要目标用户

- 容易拖延、启动困难的人
- 容易被手机、娱乐、即时快感打断的人
- 普通番茄钟、待办清单、习惯打卡效果不佳的人
- 想将自控方法做成系统化流程，而非依赖“鸡血”的人

### 5.2 非目标用户

- 只想要一个极简倒计时器的人
- 只想统计学习时长、不想执行协议的人
- 期待软件自动替代个人判断的人

---

## 6. 产品设计原则

### 6.1 协议优先，而非数据优先

统计数据服务于协议执行，而不是反过来。

例如：

- “今天学了多少分钟”不是最重要的
- “预约链是否被履约”“主链是否延续”“判例是否被谨慎使用”才是核心

### 6.2 规则明确，而非模糊鼓励

产品界面应尽量避免：

- “加油！”
- “再坚持一下！”
- “你可以做到！”

这些话不是核心价值。Protocol 应以明确协议、清晰决策、强约束反馈为主。

### 6.3 用户做裁判，软件做执行器

软件不替用户判断“刷两分钟视频算不算违规”。

软件负责：

- 要求用户正式判定
- 保存判定
- 在未来保持判例一致性

### 6.4 允许失败，但不允许模糊

失败可以发生：

- 链条可以断
- 预约可以失败
- 节点可以熄灭

但每次失败都应该留下清晰记录，而不是被吞没在“算了，下次注意”中。

### 6.5 桌面工具气质

界面风格：

- 简洁
- 冷静
- 克制
- 稍带“控制台 / 协议面板”气质
- 不做花哨游戏化

## 7. V1 产品范围

V1 只做：

### 7.1 CTDP 主链系统

- 创建主链
- 查看主链
- 进入链详情
- 启动一次正式专注任务
- 任务完成后，链长度 +1
- 任务失败后，用户必须判定：
  - 链断裂并清零
  - 当前行为允许，写入判例

### 7.2 预约链系统

- 发起“预约启动”
- 设定默认预约倒计时，例如 15 分钟
- 到期后要求用户履约进入主链任务
- 若未履约，也必须判定：
  - 预约失败
  - 当前情形写入预约判例

### 7.3 判例库

- 每条主链有独立判例库
- 判例至少包含：
  - 名称
  - 说明
  - 作用域：主链 / 预约链
  - 创建时间
  - 备注
- 在链详情页可查看已有判例

### 7.4 历史记录

记录：

- 每一次预约启动
- 每一次正式任务
- 成功 / 失败 / 判例化
- 相关时间点
- 当时关联的链

### 7.5 基础仪表盘

首页展示：

- 当前主链数量
- 当前最长主链
- 今日已完成任务数
- 最近一次专注任务
- 最近一次预约记录

### 7.6 RSIP 占位模块

V1 中只保留：

- 导航入口
- 空白介绍页或“即将到来”页
- 不实现国策树业务逻辑

## 9. 关键交互流程

### 9.1 主链任务流程

1. 用户进入某条主链
2. 点击“开始正式任务”
3. 软件显示任务进行中
4. 用户在现实中执行专注任务
5. 到达目标时间后，用户选择：
   - 成功完成
   - 中途失败
6. 若成功：
   - 当前链长度 +1
   - 写入成功记录
7. 若失败：
   - 进入判定界面
   - 二选一：
     - 违规：链清零
     - 判例化：添加判例，不清链

### 9.2 预约启动流程

1. 用户点击“预约启动”
2. 选择目标链
3. 预约倒计时开始
4. 到期时进入履约界面
5. 用户选择：
   - 立即进入正式任务
   - 未履约
6. 若未履约：
   - 进入预约判定界面
   - 二选一：
     - 预约失败
     - 当前情况写入预约判例

## 11. 术语表

- **主链**：正式任务执行链
- **预约链**：延时启动的承诺链
- **链节点**：一次成功履行协议的记录
- **链断裂**：因违规导致当前链长度归零
- **判例**：某次争议行为被正式认定为以后永久允许的规则记录
- **判例化**：将争议行为写入判例库
- **正式任务**：一次触发主链后必须完成的专注任务
- **预约启动**：先承诺稍后启动，再在约定时限内进入正式任务
- **国策 / 定式**：RSIP 中一个明确、低阻力、可执行的小规则
- **国策树**：定式之间的依赖结构

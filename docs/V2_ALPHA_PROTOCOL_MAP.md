# Protocol V2 Alpha 功能映射

本文说明当前软件功能如何对应 CTDP 与 RSIP 方法论。Protocol 是行为协议工具，用于记录、执行和复盘用户自己设定的行为规则；它不是医疗诊断、心理治疗或健康干预软件。

## CTDP：链式时延协议

### 主链

对应 CTDP 的 Main Chain。用户创建一条主链后，每次启动正式任务都意味着进入该链的协议状态。

- 正式任务完成：写入 `focus_sessions.completed`，主链当前长度加一。
- 正式任务失败并裁定违规：写入 `focus_sessions.failed_reset`，主链当前长度清零。
- 正式任务失败并判例化：写入 `focus_sessions.failed_precedent`，新增一条主链判例，主链当前长度不清零。

### 判例法

对应“下必为例”原则。软件不替用户判断某个行为是否违规，而是要求用户做出正式裁决。

- 判定违规：承认本次行为破坏协议，并执行链清零。
- 判例化：承认该行为以后在同类情形中永久允许，并保存到判例库。

### 预约启动

对应“线性时延原理”。预约启动把“现在开始”的高阻力动作转化为“稍后必须开始”的低阻力承诺。

- 预约履约：写入 `reservation_sessions.fulfilled`，并进入正式任务。
- 预约失败：写入 `reservation_sessions.failed_reset`，主链长度不受影响。
- 预约判例化：写入 `reservation_sessions.failed_precedent`，新增一条预约判例。

V2 Alpha 仍保留 V1 的轻量预约链实现：记录预约履约与裁决，但暂不维护独立预约链长度。

## RSIP：递归稳态迭代协议

### 定式节点

对应 RSIP 中的“国策 / 定式”。每个定式应是明确、低阻力、可执行的小规则。

当前字段包括：

- 标题
- 执行说明
- 父节点
- 点亮状态
- 创建、点亮、熄灭时间

### 定式树

对应 RSIP 的递归回溯结构。用户可以创建根定式，也可以把新定式挂到已有定式下面。

- 根节点代表可以独立存活的基础定式。
- 子节点代表依赖父节点稳态而更容易存在的后续定式。
- 页面使用缩进树展示定式之间的依赖关系。

### 点亮、熄灭与回滚

定式点亮表示它当前处于生效状态。定式熄灭表示用户裁定该定式当前未能维持。

当一个 active 父节点熄灭时，系统会递归熄灭其所有 active 子孙节点，并写入 `rollback_child_deactivated` 事件。这对应 RSIP 的回滚机制：父节点崩塌时，依赖它的后续节点同步退回。

### 事件日志

`formula_events` 记录 RSIP 的关键事件：

- `created`：定式加入树
- `activated`：定式点亮
- `deactivated`：定式熄灭
- `rollback_child_deactivated`：因父节点熄灭触发递归回滚

Dashboard 会展示 RSIP 摘要；History 会把 CTDP 与 RSIP 事件合并成统一协议时间线。

## V2 Alpha 边界

本版本实现的是最小可用 RSIP 定式树，不包含：

- 国策组容错
- 强化升级型定式
- 独立预约链长度
- 二次预约信号
- AI 建议
- 医疗评估、诊断或治疗建议

这些内容可以在后续阶段继续扩展，但不应破坏 V1 已有的主链、预约与判例闭环。
